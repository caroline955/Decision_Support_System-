2. Objectives of the Study
2.1. General Objective
The general objective of this project is to build an AI Chatbot system that supports students in learning Data Science while exploiting question-answering data to generate signals that assist lecturers in monitoring learning activities and making instructional decisions.
The system is oriented as a prototype Decision Support System in education, where the chatbot serves as the student interaction point, conversational data is stored and processed, and lecturers receive analytics and alerts to support early observation and intervention when necessary.
 
2.2. Specific Objectives
To achieve the general objective, the project focuses on the following specific objectives:
To develop a web application with three main roles: Admin, Teacher, and Student, in which each role has its own interface, functions, and data access scope.
To build the backend using FastAPI, organized by routers and services, providing APIs for authentication, academic data management, chatbot processing, retrieval, analytics, alerts, and admin statistics.
To build the frontend using React, Vite, TypeScript, and Tailwind CSS, organized by layouts and pages for each role to ensure a clear and role-appropriate user interface.
To design a MySQL database for managing users, courses, classes, schedules, lessons, chat sessions, chat messages, learning analytics, and teacher alerts.
To develop a chatbot that allows students to ask questions related to Data Science and stores chat history so that students can review previous interactions and the backend can use the data for analysis.
To implement a keyword retrieval mechanism for finding lessons related to students’ questions. In the current version, the system uses keyword scoring instead of a vector database to prioritize simplicity, explainability, and suitability for a local demo environment.
To implement an analytics mechanism based on the number of times a student repeatedly asks about the same topic within a certain period. The system calculates repeat score and stores or updates the result in the learning_analytics table.
To implement an alert mechanism for lecturers when a student repeatedly asks about the same topic. Specifically, when a student asks about the same topic three or more times within seven days, the system generates a repeat-question alert so that the lecturer can observe the issue and consider providing support.
To provide an admin dashboard that displays system-level statistics, including users, teachers, students, courses, classes, lessons, enrollments, chat sessions, chat messages, and alerts.
To test the system from multiple perspectives, including functional testing, API testing, role-based access control testing, data consistency testing, chatbot flow testing, retrieval testing, analytics and alert testing, error handling testing, and basic performance testing.
 
3. Research Objects and Scope
3.1. Target Users
The system is designed for three main user groups. The first user group is Admin. Admins are responsible for managing general system data. They can log in, view statistical dashboards, manage teachers, students, courses, classes, and perform administrative tasks such as recomputing analytics.
The second user group is Teacher. Teachers directly teach and monitor classes. They can view assigned classes, view class details, view students in a class, view or create schedules, view learning analytics in class details, view AI alerts, and mark alerts as read.
The third user group is Student. Students use the chatbot to support their learning. They can log in, view enrolled classes, view schedules, ask the chatbot, view retrieved related lessons, review chat history, and view personal analytics on the dashboard.
 
3.2. System Scope
The system is implemented as a web application running in a localhost demo environment. The backend runs with FastAPI and Uvicorn, the frontend runs with React and Vite, and the database uses MySQL through XAMPP or an equivalent MySQL environment.
The data used in this project consists of internal data and simulated seed data for demonstration, functional testing, and illustrating the analytics and alert mechanisms. The learning content focuses on the Data Science subject. The lessons table can be created from seed data or populated through a PDF ingestion script to support keyword retrieval.
In the current version, retrieval is implemented using keyword scoring, without a vector database or embedding-based retrieval. This approach is suitable for the prototype scope because it is simple to implement, explainable, and directly testable. The system supports multiple LLM providers, such as mock, Gemini, OpenAI, or Ollama, depending on the configuration. If the mock provider is used, the chatbot is evaluated only in terms of functional flow and not in terms of actual AI response quality.
In terms of decision support, the current system focuses on repeat-question alerts. The backend analyzes how many times a student repeatedly asks about the same topic within seven days, calculates repeat score, and generates an alert for the lecturer when the threshold is reached. More advanced metrics, such as Engagement Score based on SAW/MCDM, topic diversity, engagement ranking, top topics chart, or activity trend chart, are not considered completed functions in the current implementation.
 
3.3. Out of Scope
The project does not deploy the system to a production cloud environment. Issues such as HTTPS, production logging, monitoring, load balancing, advanced rate limiting, and in-depth security auditing are outside the current scope.
The project does not integrate with real Learning Management Systems such as Moodle, Canvas, or Google Classroom. Users, classes, schedules, lessons, and conversations are created as simulated seed data.
The project does not fine-tune an LLM or build a new language model. The system only integrates available providers or a mock provider through configuration. If a real provider such as Gemini, OpenAI, or Ollama is used, the response quality depends on the provider, prompt, retrieval context, and testing time.
The project does not evaluate the system using real student data and does not claim to predict final grades. The current analytics serve only as reference signals for lecturers and do not replace professional judgment or official academic assessment.
The current version also does not implement a vector database. Future work may include embedding retrieval, vector search, Engagement Score based on SAW/MCDM, topic diversity, engagement ranking, top topics chart, activity trend chart, low-activity alerts, and narrow-topic alerts.
 
4. Methodology
The project is conducted through the analysis, design, implementation, and testing of a full - stack software system. First, system requirements are analyzed based on user actors. The three main user groups, Admin, Teacher, and Student, are identified together with their corresponding functions. Actor-based analysis helps separate permissions, interfaces, and usage flows for each user group.
Second, a relational database is designed to store the main entities of the system, including users, courses, classes, schedules, lessons, chat sessions, chat messages, learning analytics, and teacher alerts. The database design provides the foundation for managing academic data, storing conversation history, and supporting dashboards and analytics.
Third, the overall system architecture is designed according to the frontend-backend-database model. The frontend is built with React, Vite, TypeScript, and Tailwind CSS. The backend is built with FastAPI. The database uses MySQL. This architecture separates the user interface, business logic, and data storage layers, improving maintainability and extensibility.
During implementation, the backend is organized by routers and services to separate functional groups such as authentication, user management, academic data, chatbot, analytics, alert, and admin statistics. The frontend is organized by layouts and pages for each role to ensure consistent and role-appropriate user interfaces.
For the chatbot function, the system implements a processing flow that includes checking the Student role, creating or retrieving a chat session, detecting the topic using keyword heuristics, retrieving related lessons, calling the LLM provider, saving the student’s message and the bot’s response, and returning the result to the frontend. Retrieval is implemented using keyword scoring, where keywords appearing in the title have a higher weight than keywords appearing in the content.
For the analytics and alert components, the system implements a mechanism to detect repeated questions about the same topic within seven days. From chat messages, the system calculates repeat score, updates the learning_analytics table, and creates a teacher_alert when the number of repeated questions reaches the threshold. This mechanism is the main decision-support component in the current version.
Finally, the system is tested through multiple test groups, including functional testing, role-based access control testing, API testing, chatbot flow testing, retrieval testing, analytics and alert testing, data consistency testing between the user interface and database, error handling testing, and basic performance testing. These tests demonstrate that the system is not only a user interface but also has a complete data flow from frontend to backend and database.
 
5. Contributions of the Project
The project makes the following main contributions:
It builds a full-stack prototype with three roles: Admin, Teacher, and Student, in which each role has its own interface and functions.
It integrates a learning chatbot with conversation history, allowing students to ask questions about Data Science and review previous chat sessions.
It implements a keyword-scoring mechanism to retrieve lessons related to students’ questions, linking chatbot interaction with learning content stored in the database.
It designs a database that stores academic data, schedules, lessons, chat sessions, chat messages, learning analytics, and teacher alerts.
It implements a mechanism for detecting repeated questions about the same topic based on chat messages and question topics.
It provides AI alerts for lecturers when a student asks about the same topic three or more times within seven days, helping lecturers observe possible learning difficulties and intervene when necessary.
It provides an admin statistics dashboard that helps admins monitor users, courses, classes, lessons, chat sessions, chat messages, and alerts.
It implements and tests an end-to-end system flow, including login, role-based access control, academic data management, chatbot processing, retrieval, chat history storage, analytics, and teacher alerts.
These contributions show that the project is not limited to building a standalone question-answering chatbot. Instead, it moves toward a system that records and uses learning interaction data to support lecturers in instructional decision-making.
 

 
CHAPTER 1. THEORETICAL BACKGROUND AND RELATED WORK
This chapter presents the theoretical foundations that directly support the development of the AI Chatbot system for Data Science learning. The system is positioned as a prototype Decision Support System in education. Its main purpose is not only to answer students’ questions, but also to collect learning interaction data, analyze repeated questions by topic, and provide alerts that help lecturers observe possible learning difficulties. Therefore, this chapter focuses on the concepts that are directly connected to the implemented system: Decision Support System, Simon’s decision-making model, educational chatbots, Large Language Models, retrieval-based question answering, Learning Analytics, Early Warning, and related work.
1.1. Decision Support System
A Decision Support System, or DSS, is an information system designed to support human decision-making, especially in situations where decisions are not fully structured or cannot be completely automated (Turban et al., 2011). In education, many decisions made by lecturers belong to this category. For example, a lecturer may need to decide whether to review a topic, provide additional materials, or support a student who seems to be struggling. These decisions require professional judgment, but they can be supported by learning data.
In this project, the DSS concept is applied to the teaching context. The system does not make decisions on behalf of lecturers. Instead, it collects data from student-chatbot interactions, analyzes repeated questions by topic, and generates alerts when a student repeatedly asks about the same topic. These alerts act as signals that help lecturers observe possible learning problems and decide whether further action is needed.
A DSS usually consists of three main components: Data Management, Model Management, and User Interface. In this system, Data Management is represented by the MySQL database, which stores users, courses, classes, lessons, chat_sessions, chat_messages, learning_analytics, and teacher_alerts. Model Management is represented by simple but explainable logic, including keyword-based topic detection, keyword retrieval, repeat_score calculation, and repeat-question alert rules. User Interface is represented by the Admin, Teacher, and Student portals, where different users interact with the system according to their roles.
Therefore, the system can be considered a DSS prototype because it has the three essential elements: data, analytical logic, and an interface for decision support. However, it is important to clarify that the system does not include a complex prediction model. Its decision-support function is currently limited to detecting repeated questions and presenting alerts to lecturers.
1.2. Decision-Making Process in Teaching
Simon’s decision-making model includes four stages: Intelligence, Design, Choice, and Implementation (Simon, 1977). This model is useful for explaining how the system supports lecturers in the teaching process. The Intelligence stage is about identifying a problem. The Design stage is about analyzing the problem and preparing possible actions. The Choice stage is about selecting an appropriate action. The Implementation stage is about carrying out that action in practice.
In this project, the Intelligence stage corresponds to collecting student interaction data. When a student asks the chatbot, the system stores the chat session, message content, detected topic, and related lessons. The Design stage corresponds to analyzing that data. The backend checks how many times the student has asked about the same topic within seven days and calculates a repeat_score. The Choice stage happens when the lecturer views the generated alert and decides what to do. The Implementation stage happens when the lecturer takes action, such as reviewing the topic in class, sending additional materials, or contacting the student.
This mapping is important because it explains why the system is more than a chatbot. The chatbot is the entry point for collecting learning data, while the analytics and alert mechanism support the lecturer’s decision-making process. The system does not automate teaching decisions, but it helps lecturers identify signals that may otherwise be missed.
1.3. Chatbots in Education
An educational chatbot is a software system that allows learners to interact through a conversational interface for learning purposes. It can answer questions, provide explanations, suggest learning materials, and support students outside classroom hours. Studies on educational chatbots show that they can support learning through quick responses, mentoring, guidance, and personalization (Wollny et al., 2021).
In this project, the chatbot is used to support students in learning Data Science. Students can ask questions related to topics such as pandas, missing values, train-test split, clustering, overfitting, or confusion matrix. The chatbot helps students receive support during self-study and allows them to review previous conversations through stored chat history.
However, the chatbot in this system has another important role: it creates learning interaction data. Each student question is stored as a chat message and can be analyzed later. When a student repeatedly asks about the same topic, the system can use this pattern to generate an alert for the lecturer. Therefore, the chatbot is not only a question-answering tool, but also the starting point of the Learning Analytics and Decision Support flow.
The limitations of chatbots must also be acknowledged. A chatbot may provide incorrect or incomplete answers, especially when using a real LLM provider. The quality of the answer depends on the model, prompt, retrieval context, and learning materials stored in the database. If the system uses a mock provider, it can only be used to test the functional flow, not to evaluate real AI response quality. For this reason, the report must clearly distinguish between functional testing and chatbot response quality evaluation.
1.4. Large Language Model and LLM Provider
A Large Language Model, or LLM, is a model capable of understanding and generating natural language text. In chatbot systems, an LLM can be used to generate answers, explain concepts, and provide examples. In this project, the LLM is used in the response generation step after the student submits a question and the system retrieves related lessons.
The system is designed to support different LLM providers through configuration, including Mock, Gemini, OpenAI, and Ollama. The Mock provider is mainly used for functional testing. It allows the developer to verify whether the chatbot flow works correctly, including sending a question, receiving a response, saving chat messages, returning related lessons, and updating analytics. However, because mock responses are simulated, they cannot be used to evaluate real AI answer quality.
Gemini and OpenAI are cloud-based providers that can generate more natural answers, while Ollama can be used for local models. If a real provider is used for evaluation, the report must state the provider name, model version, testing date, prompt template, and retrieval mode. This is necessary to ensure transparency and reproducibility. Without this information, chatbot evaluation results would be difficult to verify.
1.5. Retrieval-Based Question Answering
Retrieval is the process of finding relevant documents or content based on a user’s question. In chatbot systems, retrieval helps provide context so that the answer can be more closely connected to the available learning materials. Retrieval-Augmented Generation shows the value of combining generated answers with externally retrieved knowledge (Lewis et al., 2020).
In this project, retrieval is used to find lessons related to a student’s question. The current system uses keyword retrieval rather than vector retrieval. When a student submits a question, the system tokenizes the question, removes stopwords, compares keywords with lesson titles and lesson content, calculates a score, ranks candidate lessons, and returns the top related lessons.
The scoring formula used in the current system is:
score = 3 × title_overlap + 1 × content_overlap
This means that a keyword appearing in the lesson title has higher importance than a keyword appearing only in the lesson content. This design is reasonable because the title usually represents the main topic of the lesson. For example, if a student asks “What is pandas groupby used for?”, lessons with keywords such as “pandas” or “groupby” in the title or content will receive a higher score.
Keyword retrieval is chosen because it is simple, explainable, and suitable for a local prototype. It does not require embeddings, a vector database, or additional infrastructure. However, it also has limitations. It may fail when students use different wording, synonyms, or paraphrased questions. It also depends heavily on the quality of lesson titles and content. Therefore, vector retrieval or embedding-based search is considered a future improvement, not a completed feature of the current system.
1.6. Learning Analytics and Early Warning
Learning Analytics is the process of collecting, measuring, analyzing, and reporting data about learners and learning contexts in order to understand and improve learning and teaching (Siemens & Long, 2011). In this project, the main source of learning analytics data is chatbot interaction. Each time a student asks a question, the system stores the message, timestamp, chat session, detected topic, and related lessons.
The current analytics mechanism focuses on repeated questions by topic. If a student asks about the same topic multiple times within seven days, this may indicate that the student is interested in or struggling with that topic. The system uses this pattern to calculate repeat_score and generate teacher alerts.
The repeat_score is calculated as:
repeat_score = min(1.0, n / 5.0)
In this formula, n is the number of times a student asks about the same topic within seven days. If n = 0, repeat_score = 0 and no alert is generated. The value 5 is used as an experimental threshold: when a student asks about the same topic five or more times within seven days, repeat_score reaches 1.0. This parameter can be adjusted in a real deployment.
Early Warning in education refers to mechanisms that detect early signs that a student may need support. Course Signals at Purdue is an example of using learning data to support early intervention (Arnold & Pistilli, 2012). In the current system, Early Warning is implemented in a simple and explainable form: repeat-question alert. If a student asks about the same topic three or more times within seven days, the system creates a teacher_alert with alert_type = repeat_question. If the number of repeated questions is three or four, the severity is medium. If it is five or more, the severity is high.
This alert does not prove that the student is weak or does not understand the topic. It only provides a signal for the lecturer to observe further. This is an important limitation. The system currently does not measure full engagement, does not predict final grades, and does not implement advanced models such as Engagement Score SAW/MCDM or topic diversity. These features are future work.

 
1.7. Related Work
Related studies can be grouped into five main areas: educational chatbots, retrieval/RAG, Learning Analytics, Early Warning Systems, and DSS in education. Educational chatbot studies show that chatbots can support students through question answering, guidance, and automated feedback. Wollny et al. (2021) reviewed chatbot applications in education and identified roles such as communication support, mentoring, and personalization. However, many chatbot systems focus mainly on answering students’ questions and do not deeply exploit conversational data for lecturer decision support.
Retrieval and RAG studies focus on retrieving relevant context to improve answer generation. Lewis et al. (2020) introduced Retrieval-Augmented Generation as a method that combines generative language models with retrieved knowledge. Compared with full RAG systems, this project only implements keyword retrieval. It does not include embeddings, vector databases, or strong paraphrase handling. However, keyword retrieval is appropriate for the current prototype because it is transparent and easy to test.
Learning Analytics studies focus on using learning data to understand and support the learning process. Siemens and Long (2011) emphasized the importance of analytics in improving education through data-informed insight. In this project, Learning Analytics is implemented in a narrow but practical way: chatbot interaction data is used to calculate repeat_score and support teacher alerts.
Early Warning Systems aim to detect students who may need support before problems become serious. Arnold and Pistilli (2012) introduced Course Signals as an example of using learning analytics for early intervention. The current system does not implement a full early warning model. It focuses only on repeat-question alerts, which are explainable and directly connected to chatbot interaction data.
DSS in education focuses on supporting data-informed decisions by lecturers or administrators. From the DSS perspective, a system should include data, processing logic, and an interface for decision support (Turban et al., 2011). The system in this project satisfies these elements at the prototype level: MySQL stores learning data, keyword retrieval and repeat_score provide processing logic, and dashboards plus teacher alerts present information to users.

CHAPTER 2. SYSTEM REQUIREMENT ANALYSIS
2.1. Problem Description
The proposed system is an AI Chatbot System for Data Science learning, designed as a Decision Support System prototype for teaching and learning support. The system focuses on three main user groups: Admin, Teacher, and Student.
In a Data Science course, students often face difficulties with technical topics such as Python programming, pandas, data visualization, linear regression, classification, and model evaluation. When students do not fully understand a concept, they may repeatedly ask similar questions. However, in a traditional teaching environment, teachers may not immediately know which students are struggling or which topics are being asked about most frequently. The system addresses this problem by allowing students to ask questions through a chatbot. Each chatbot interaction is stored as chat history. The system also detects the topic of the student’s question, retrieves related lessons using keyword-based retrieval, generates an answer using an LLM provider, and stores both the student question and the bot response. From the teacher’s perspective, the system provides learning analytics and AI alerts. When a student asks about the same topic repeatedly within a recent period, the system can generate an alert for the teacher. This allows the teacher to identify possible learning difficulties and decide whether to review a topic, contact a student, or provide additional learning materials. From the admin’s perspective, the system provides an overview dashboard showing statistics such as the number of users, teachers, students, courses, classes, lessons, enrollments, chat sessions, chat messages, and alerts. Admins can also manage teachers, students, courses, and classes.
Therefore, the system is not only a chatbot for answering questions. It is a prototype Decision Support System in which chatbot interactions become learning behavior data, and that data is transformed into simple decision-support signals for teachers and administrators.

The system has three main human actors: Admin, Teacher, and Student. In addition, the system contains internal service actors such as the Retrieval Service, LLM Service, Analytics Engine, Alert Engine, and Database.

Admin is the high-level account that manages teacher accounts and assigns classes. Teacher accounts operate within the scope assigned by Admin and are responsible for monitoring students, schedules, chat history signals, analytics and alerts in their classes.

CHAPTER 3. SYSTEM DESIGN
3.1. System Architecture
The DS Chatbot System is designed as a full-stack web application with a frontend-backend-database architecture. The system separates the user interface, business logic, service modules, and data storage into different layers to improve maintainability and scalability.
The system consists of four main layers:
User Layer
Frontend Layer
Backend Layer
Data Layer
The User Layer includes three human actors: Admin, Teacher, and Student. Each actor accesses a different role-based portal. Admin uses the system to manage users, courses, and classes, and to view overall statistics. Teacher uses the system to view assigned classes, schedules, learning analytics, and AI alerts. Student uses the system to view enrolled classes, schedules, ask the chatbot, and review chat history.
The Frontend Layer is implemented using React, Vite, TypeScript, and Tailwind CSS. It includes role-based layouts, pages, authentication context, route guards, and an API client. The frontend sends HTTP requests to the backend and displays the returned data to the user.
The Backend Layer is implemented using FastAPI. It contains routers for authentication, users, academic management, chatbot, analytics, and admin statistics. The backend also contains service modules for retrieval and LLM provider integration. It is responsible for authentication, authorization, business logic, chatbot processing, analytics recomputation, and alert generation.
The Data Layer uses MySQL as the relational database and SQLAlchemy as the ORM. The database stores users, courses, classes, schedules, lessons, chat sessions, chat messages, learning analytics, and teacher alerts.

Figure 3.1. System Architecture of DS Chatbot System
This architecture is suitable for a prototype Decision Support System because each layer has a clear responsibility. The frontend focuses on user interaction, the backend handles processing and decision-support logic, and the database stores learning and system data.

3.4. Sequence Diagram Design
This section presents the main sequence diagrams of the system. These diagrams describe the interaction order between frontend pages, backend routers, services, and the database.
3.4.1. Login Sequence
The login sequence describes how a user logs into the system and is redirected based on role.

Figure 3.5. Login Sequence Diagram
The login process is as follows:
The user opens the login page.
The user enters email and password.
The frontend sends a login request to POST /api/auth/login.
The backend checks whether the email exists.
The backend verifies the password using bcrypt.
The backend checks whether the account is active.
The backend creates a JWT access token.
The backend returns the token and user information.
The frontend stores the token and user data in localStorage.
The frontend redirects the user based on role:
○  	Admin to /admin
○  	Teacher to /teacher
○  	Student to /student
If the email or password is incorrect, the backend returns an authentication error. If the account is inactive, the backend rejects the login request.
3.4.2. Student Ask Chatbot Sequence
The chatbot sequence is one of the most important flows of the system. It shows how a student question is processed and transformed into a chatbot response.

Figure 3.6. Student Ask Chatbot Sequence Diagram
The process is as follows:
The student opens the AI Assistant page at /student/chat.
The student enters a Data Science question.
The frontend sends the request to POST /api/chat/ask.
The backend checks that the current user role is student.
If no session_id is provided, the backend creates a new chat session.
If a session_id is provided, the backend verifies that the session belongs to the current student.
The backend detects the topic of the question using keyword rules.
The backend calls the Retrieval Service to search for related lessons.
The Retrieval Service tokenizes the question, removes stopwords, searches candidate lessons, computes scores, and returns the top related lessons.
The backend saves the student message into chat_messages.
The backend calls the LLM Service with the question and retrieved lesson contexts.
The LLM Service returns the chatbot answer.
The backend saves the bot message into chat_messages.
The backend returns the answer, topic, related lessons, and response time to the frontend.
The frontend displays the chatbot answer and related lesson references.
This sequence demonstrates how the system combines chatbot interaction, retrieval, and data storage in one learning support flow.
3.4.3. Analytics Recompute and Alert Sequence
The analytics recompute sequence describes how the system uses chat data to generate decision-support signals for teachers.

Figure 3.7. Analytics Recompute and Alert Sequence Diagram
The process is as follows:
Admin calls POST /api/analytics/recompute.
The backend checks that the current user is admin.
The system retrieves student messages from the recent 7-day period.
The system groups messages by student_id and topic_id.
The system counts how many times each student asked about each topic.
If the count is lower than 3, the system skips alert generation.
If the count is 3 or higher, the system finds the student’s class.
The system finds the teacher assigned to that class.
The system calculates repeat score using the formula: repeat_score = min(1.0, n / 5.0)
The system inserts or updates the corresponding row in learning_analytics.
The system creates a teacher alert with alert type repeat_question.
The alert severity is set as:
○  	medium if the count is from 3 to 4
○  	high if the count is 5 or more
The teacher can later view the alert on the AI Alerts page.
This sequence shows the decision-support mechanism of the system. The system does not predict final grades. Instead, it provides early signals based on repeated learning questions.

CHAPTER 4. DECISION SUPPORT MECHANISM
4.1. Overview of the Decision Support Mechanism
The DS Chatbot System is designed not only as a question-answering chatbot, but also as a Decision Support System prototype for teaching support. The main purpose of the system is to transform student chatbot interactions into simple learning signals that teachers can use for observation and intervention.
In the current implementation, the decision support mechanism focuses on repeated-question behavior. When a student asks about the same topic many times within a recent period, the system considers this behavior as a possible sign that the student may be struggling with that topic. The system then computes learning analytics and generates alerts for the teacher.
The main decision support flow is:
Student asks a question
-> The question is stored as chat data
-> The system detects the question topic
-> The system retrieves related lessons
-> The chatbot generates an answer
-> The analytics engine recomputes learning indicators
-> The alert engine detects repeated-question behavior
-> The teacher receives an AI alert
-> The teacher decides whether to review the topic or support the student

Figure 4.1. Decision Support Flow of DS Chatbot System
This flow follows the logic of a Decision Support System. The database stores learning behavior data, the backend processes that data using simple rules and scoring mechanisms, and the user interface presents the results to teachers and admins. The system does not make final decisions automatically. Instead, it provides signals that support human decision-making.
4.2. Decision Support Model in the System
The decision support model of the system can be explained through three main dimensions: data, model, and interface.
DSS Component
Implementation in the System
Description

Admin dashboard, Teacher class detail, Teacher AI alerts, Student dashboard
Displays information to different actors according to their roles.

In this system, the chatbot plays an important role as the interaction point where learning behavior data is generated. Every student question can become a data point for later analysis. The retrieval module connects the student’s question with related lessons, while the analytics and alert modules transform repeated-question behavior into teacher-facing alerts.
The teacher remains the final decision-maker. The system only suggests that a student or topic may need attention. The teacher can then decide whether to:
●  	Review the topic in the next class.
●  	Contact the student directly.
●  	Add more learning materials.
●  	Give additional examples or exercises.
●  	Monitor the student’s future activity.
Therefore, the system should be understood as a decision-support prototype, not an automatic decision-making system.
4.3. Chatbot Interaction as Learning Behavior Data
In the system, chatbot interaction is the main source of learning behavior data. When a student asks a question, the system does not only return an answer. It also stores the question, the detected topic, the retrieved lesson reference, the bot response, and the response time.

This is the first step of the decision support mechanism. Without storing chatbot interactions, the system would only be a simple chatbot and would not be able to support teacher decision-making.
4.4. Topic Detection Mechanism
The topic detection mechanism is used to assign a question topic to a student message. In the current implementation, the system uses a keyword-based heuristic approach. This means that the backend checks whether the student’s question contains specific keywords related to predefined Data Science topics.
The detected topic can later be used for analytics and alert generation. For example, if a student repeatedly asks questions related to pandas, the system can detect that the student may be struggling with the pandas topic.

Figure 4.3. Topic Detection Mechanism
Example topic groups in the current system include:
Topic
Example Keywords

python
python, list, function, loop
visualization
matplotlib, chart, plot, histogram
linear regression
regression, linear regression, prediction
classification
classification, confusion matrix, accuracy

The topic detection process can be summarized as follows:
Student question
-> Normalize text
-> Check keyword rules
-> Match question with predefined topic
-> Assign topic_id to chat message
 
If the question does not match any keyword rule, the topic may be null. In that case, the message can still be stored and answered by the chatbot, but it may not be used for repeated-topic alert generation.
The keyword-based method is simple and explainable. However, it has limitations. It may fail when a student uses different words to describe the same concept. For example, a question about “splitting training and testing data” may not be detected correctly if the keyword rule only checks for “train-test split”. This limitation can be improved in the future by using embedding-based topic detection or text classification.
4.5. Related Lesson Retrieval Mechanism
The retrieval mechanism is used to find learning materials related to the student’s question. In the current version, the system does not use a real vector database. Instead, it uses keyword-based scoring over the lessons table.
The retrieval mechanism is designed to be simple, transparent, and explainable. This is suitable for a local prototype because the teacher and developer can understand why a lesson is selected.

Figure 4.4. Keyword-based Lesson Retrieval Pipeline[12] 
The retrieval pipeline is:
Student question
-> Tokenization
-> Stopword removal
-> Candidate lesson search
-> Score calculation
-> Ranking
-> Return top 3 related lessons
The scoring formula is:
score = 3 × title_overlap + 1 × content_overlap
Where:
Term
Meaning
title_overlap
Number of question tokens appearing in the lesson title.
content_overlap
Number of question tokens appearing in the lesson content.
score
Final retrieval score of a lesson.

The title has a higher weight because a keyword appearing in the lesson title is usually more important than a keyword appearing only in the lesson content. For example, if the question is “What is pandas groupby used for?”, a lesson with the title “Data Aggregation with Pandas GroupBy” should be ranked higher than a lesson that only mentions “groupby” once in the content.
The retrieval result is returned as lesson_refs, which may include:
●  	Lesson ID.
●  	Lesson title.
●  	Retrieval score.
These lesson references are displayed in the student chatbot interface so that the student can see which learning materials are related to the answer.

4.14. Limitations of the Current Decision Support Mechanism
The current decision support mechanism is intentionally simple and suitable for a local prototype. It has several limitations:
The system uses keyword-based topic detection, so it may not understand paraphrased questions.
The retrieval module uses keyword scoring, not semantic vector search.
The analytics mechanism mainly focuses on repeated-question behavior.
The system does not implement a full Engagement Score based on SAW/MCDM in the current code.
Topic diversity is not fully implemented as a current feature.
Engagement ranking is not implemented as a current feature.
Top topics chart and activity trend chart are not treated as completed functions.
The system does not predict student grades.
The system does not automatically decide teacher actions.
The current evaluation uses simulated data rather than real classroom data.
These limitations should be clearly stated in the report to avoid overclaiming. The system should be positioned as a prototype DSS that provides early behavioral signals, not as a complete learning analytics or prediction platform.
4.15. Future Extension of the Decision Support Mechanism
The current decision support mechanism can be extended in several directions.
4.15.1. Vector-based Retrieval
The keyword retrieval module can be upgraded to embedding-based retrieval using a vector database such as Chroma, Pinecone, or pgvector. This would help the system understand semantic similarity and paraphrased questions better.
4.15.2. Engagement Score
A future version can implement a complete Engagement Score using multiple indicators such as session activity, repeat behavior, and topic diversity. The score can be designed as a weighted scoring model, but the weights should be validated with real data or teacher feedback.
4.15.3. Topic Diversity
The system can calculate how diverse a student’s questions are. If a student asks many questions but only about one topic, the system may generate a narrow-topic alert.
4.15.4. Low Activity Alert
The system can detect students who have no chatbot sessions or very low interaction within a recent period. This can help teachers identify passive students.
4.15.5. Teacher-adjustable Thresholds
Teachers can be allowed to adjust alert thresholds. For example, one teacher may want alerts after 3 repeated questions, while another may prefer alerts after 5 repeated questions.
4.15.6. Personalized Learning Recommendation
The system can recommend lessons, exercises, or review materials based on each student’s chat history and repeated topics.
These extensions can improve the decision support capability of the system, but they are not part of the completed current implementation.
4.16. Chapter Summary
This chapter presented the decision support mechanism of the DS Chatbot System. The system uses chatbot interactions as learning behavior data. Student questions are stored, topics are detected, related lessons are retrieved, and chatbot answers are generated. The stored chat data is then used by the analytics engine to detect repeated-question behavior.
The current analytics mechanism computes repeat scores based on the number of times a student asks about the same topic in a recent 7-day period. If the number of repeated questions reaches the threshold, the system generates a teacher alert. The alert severity is medium for 3-4 repeated questions and high for 5 or more repeated questions.
The system supports teacher decision-making by showing class analytics and AI alerts. However, the teacher remains the final decision-maker. The system does not predict grades and does not make automatic decisions about student performance.
The chapter also discussed admin decision support, student support, explainability, current limitations, and future extensions. Overall, the decision support mechanism is simple, explainable, and consistent with the current implementation of the system.

5.8. Retrieval Implementation
The Retrieval Service uses keyword-based scoring. It does not use a real vector database in the current version.
Figure 5.16. Keyword Retrieval Implementation[28] 
The retrieval pipeline is:
Question
-> Tokenization
-> Stopword removal
-> Candidate lesson search
-> Score calculation
-> Ranking
-> Return top-k lessons
 
The scoring rule is:
score = 3 × title_overlap + 1 × content_overlap
 
This means that a keyword found in the lesson title has a higher weight than a keyword found only in the lesson content. This design is simple but explainable.
Example:
Question: "What is pandas groupby used for?"
 
Tokens: pandas, groupby
 
Possible related lesson:
"Data Aggregation with Pandas GroupBy"
 
Reason:
The tokens appear in the lesson title, so the lesson receives a higher score.
 
The system returns up to the top 3 related lessons. These lessons are used as context for the chatbot answer and are also displayed to the student as lesson references.
This retrieval implementation is suitable for a prototype because it is easy to run locally and does not require embedding APIs or vector database infrastructure. However, it may not handle semantic similarity or paraphrased questions well.

5.10. Analytics and Alert Implementation
The Analytics and Alert module is implemented in the Analytics Router. It provides APIs for personal analytics, class analytics, analytics recomputation, viewing alerts, and marking alerts as read.
Figure 5.18. Analytics and Alert Module Implementation [30] 
The main analytics recomputation endpoint is:
POST /api/analytics/recompute
 
This endpoint is admin-only. It analyzes recent student messages and detects repeated-question behavior.
The recomputation process is:
Get recent student messages
-> Group by student_id and topic_id
-> Count repeated questions
-> Skip groups with count < 3
-> Find student class and teacher
-> Calculate repeat_score
-> Update learning_analytics
-> Create teacher_alert if needed
 
The repeat score formula is:
repeat_score = min(1.0, n / 5.0)
 
Where n is the number of times a student asked about the same topic in the recent analysis period.
The alert severity rule is:
Condition
Alert Type
Severity
3 <= n < 5
repeat_question
medium
n >= 5
repeat_question
high

The generated alerts are stored in the teacher_alerts table. Teachers can view them through:
GET /api/alerts
 
Teachers can mark an alert as read using:
PATCH /api/alerts/{alert_id}/read

This implementation supports the decision support objective of the system. The system detects repeated-question behavior and provides teacher-facing alerts, but the teacher remains the final decision-maker.
In addition to the admin-only recomputation endpoint, the current system also checks repeated-question conditions immediately after each student question is submitted through /api/chat/ask. This real-time alert check is handled by the alert service function maybe_create_repeat_alert(), so repeated-question alerts can be created or updated without waiting for an Admin to manually run analytics recomputation.
5.11. PDF Ingest Implementation
The system provides scripts for importing lesson content into the lessons table so that the Retrieval Module can search learning materials when students ask questions. The main script is scripts/ingest_pdf.py, which extracts text from a PDF file, splits content into sections or chapters, cleans the extracted text, and stores the result as lesson records linked to a selected course.
In the demo environment, the original Nathan George PDF file could not be used reliably because the file produced an end-of-file extraction error. Therefore, the demonstration used a smaller Data Science learning material file, named sach.pdf or SÁCH.txt depending on the local copy, which was successfully imported into the lessons table. The imported content produced approximately 15 lesson records, each containing about 1,500 to 3,000 characters of learning content.
The project also supports manual lesson seeding through seed data when PDF ingestion is not available. This fallback approach ensures that the chatbot can still retrieve related lessons during demonstration and testing.
The ingest process can be summarized as follows:
Read learning material file -> Extract text -> Split content by heading or section -> Clean text -> Create lesson records -> Store lessons in the database -> Use lessons for keyword-based retrieval
Example command:
cd backend
 python -m scripts.ingest_pdf "../sach.pdf" --course DS101 --replace
The --replace option allows the script to replace existing imported lessons for the selected course. This is useful during development when the lesson data needs to be refreshed. Although the current retrieval method uses keyword scoring, the embedding_id field is kept in the lessons table so that the system can be extended with vector-based retrieval in the future.

CHAPTER 6. TESTING
6.1. Testing Objectives
Testing is an important phase in the development of the DS Chatbot system. The main purpose of this phase is to verify that the system satisfies the functional and non-functional requirements defined in the analysis and design stages. Through testing, the development team can evaluate whether the system operates correctly, whether data is processed consistently, and whether each user role can access only the functions that belong to its permission scope.
For this project, testing does not only focus on basic web application functions such as login, user management, class management, and authorization. Since the system is designed as an AI chatbot and decision support system for Data Science learning, special attention is given to the Retrieval Module, LLM integration, learning analytics, and automatic teacher alerts. These functions are considered the core of the system because they directly support students in learning and help teachers identify common learning difficulties.
The testing process also aims to detect implementation errors early and ensure that the system is stable enough for demonstration and academic evaluation. In addition, testing is used to verify that Vietnamese text is displayed correctly, forms are validated properly, and the interface supports the main workflows of Admin, Teacher, and Student.
The selected test cases are derived from the original functional requirement groups: managing accounts, managing classes and courses, scheduling classes, asking the AI chatbot, storing chat history, detecting repeated questions, notifying teachers, and tracking student learning frequency. This ensures that the testing chapter does not test unrelated functions but focuses on the core educational DSS behavior required by the project.

6.7. Testing Limitations
Although the testing process covered the main business workflows, several limitations remain. First, the system does not currently include automated test suites such as Pytest or Jest. Therefore, regression testing still depends on manual execution and may take more time when the system grows larger.
Second, performance testing was only conducted at a lightweight level using a small number of requests. The system has not been tested using professional load testing tools such as JMeter or k6. As a result, the current testing results are suitable for academic demonstration but are not enough to guarantee production-level scalability.
Third, security testing was limited to authentication, authorization, password hashing, and basic access control validation. The system has not been tested through professional penetration testing. SQL injection, XSS, CSRF, and other security risks should be tested more thoroughly in future development.
Finally, LLM-based responses are non-deterministic. This means that it is difficult to assert the exact response text in a traditional test case. Therefore, chatbot testing focuses mainly on whether the system returns a valid response, whether relevant lesson context is retrieved, and whether the interaction is saved correctly.
6.8. Conclusion
The testing results show that the DS Chatbot system satisfies the main acceptance requirements of the project. The system successfully supports the three main user roles: Admin, Teacher, and Student. Core workflows such as authentication, user management, class assignment, chatbot question answering, learning analytics generation, and automatic teacher alerts operate correctly.
The testing process also identified twelve implementation defects, all of which were fixed and verified again. This improved the stability and reliability of the system before final demonstration.
Overall, the DS Chatbot system is stable enough for academic presentation and project defense. Future improvements should focus on automated testing, continuous integration, larger-scale performance testing, deeper security validation, and more deterministic evaluation of chatbot responses.
